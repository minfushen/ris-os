# 风控课程 · 第 03 讲：特征宽表、模型产物与微服务纳管

> 目标：把**模型虾**在机构里常见的落地路径讲清楚：特征从哪来、模型文件长什么样、怎么**注册成服务**统一纳管，以及和**数据湖 T+1**、**分析虾**的分工。

---

## 1. 从「多表关联」到「特征宽表」

| 环节 | 说明 |
|------|------|
| **明细层** | 申请、审批、客户、借据等多表（第 01 讲） |
| **加工** | JOIN、聚合、窗口函数 → 每笔申请（或每个客户）**一行** |
| **特征宽表** | 列 = 特征（数值/类别）；行 = 样本（训练或线上推理） |

**模型虾**训练与线上打分，通常都要求：**线上推理用的表结构与训练宽表一致**（字段名、缺失值处理、编码方式），否则有**线上线下不一致**风险。

---

## 2. 常见模型产物格式

| 格式 | 常见场景 | 备注 |
|------|----------|------|
| **PKL / Pickle**（如 sklearn） | Python 栈快速上线 | **版本强绑定** Python、sklearn 版本；跨语言差，需统一运行环境 |
| **PMML** | 需与 Java 生态互操作、老核心对接 | 表达能力有限，复杂模型可能不支持或需拆分 |
| **ONNX** | 跨框架部署、多种运行时 | 深度学习与传统 ML 部分场景可统一 |
| **原生格式**（XGBoost/LightGBM 等） | 树模型直接加载 | 常配合自定义预处理服务 |

**产品设计启示**：模型虾若支持「上传模型文件」，需明确 **允许格式**、**依赖版本**、**特征清单（JSON/YAML）** 一并注册，否则无法纳管。

---

## 3. 「注册成微服务」在做什么？

典型能力：

| 能力 | 说明 |
|------|------|
| **模型注册** | 模型 ID、版本、特征 schema、依赖镜像/环境 |
| **部署** | 容器化，挂载模型文件；启动 **推理 API**（同步/异步） |
| **网关纳管** | 统一鉴权、限流、路由 `/score/v1` |
| **监控** | QPS、延迟、错误率；**输入分布漂移**（可选） |

与分析虾的关系：

- **分析虾**：偏 **探索、报表、归因**（SQL + 可视化 + 自然语言总结）。  
- **模型虾**：偏 **可重复执行的打分/策略结果**（同样依赖底层数据，但输出是**分数或决策**，不是报表）。

---

## 4. 与数据湖、调度、T+1 的关系

| 事实 | 对产品的影响 |
|------|----------------|
| 特征多来自 T+1 批处理 | **实时申请**若要走模型，需 **实时特征管道**（另一套）或接受「昨日特征」 |
| 宽表延迟 | 模型服务再快，也受上游特征产出时间限制 |
| 血缘 | 特征变更 → 需触发**模型重训或版本升级** |

**一句话**：模型微服务是「**最后一棒**」；前面「取数 + 特征加工」断了，模型只能报错或打空特征。

---

## 5. 信审虾与模型的交界

- **信审虾**：单笔资料、OCR、规则、**人审辅助**；可能调用 **规则引擎** 或 **轻量模型**（如文本分类）。  
- **模型虾**：标准化评分卡 / 机器学习模型 **批量或实时** 输出。  
- 产品上可区分：**信审**偏「材料与合规」；**模型**偏「分数与策略阈值」——数据上可能共用部分字段，但**工作流与责任人**不同。

## 6. 自检清单（模型纳管）

- [ ] 训练宽表与线上一致性（字段、缺失、编码）是否有**自动化校验**？  
- [ ] 模型版本与特征版本是否**绑定发布**？  
- [ ] 推理 API 是否记录 **request_id** 与 **模型版本**，便于审计？  
- [ ] PKL 部署是否**锁定** Python/sklearn 版本？
